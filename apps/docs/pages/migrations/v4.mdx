# Upgrade to V4

## Key changes

- Support for Vite 8
- Drops support for Vite 7
- The plugin now depends on **csp-toolkit 1.5+**. You can keep passing a plain `policy` object with kebab-case directive keys and string sources, or you can use **`definePolicy`** (recommended) for camelCase keys, keyword helpers, and stricter types.

## Required changes

To upgrade the **build tooling** only, bump `vite` to 8 and `vite-plugin-csp-guard` to 4. No CSP policy rewrites are required if you are happy with your existing `policy` object.

## Optional: the new `definePolicy` API

[csp-toolkit 1.5](https://www.npmjs.com/package/csp-toolkit) adds `definePolicy`, `mergePolicies`, and small helpers like `self`, `unsafeInline`, and scheme tokens (`data`, `blob`, etc.). The plugin’s `policy` option accepts either a legacy [`CSPPolicy`](https://www.npmjs.com/package/csp-toolkit) object or the result of `definePolicy()` (a `DefinedPolicy` with the same wire-format directives plus a `.toString()` helper).

**Before** (still supported):

```ts
csp({
  policy: {
    "script-src": ["'self'", "https://www.google-analytics.com"],
    "font-src": ["'self'", "https://fonts.gstatic.com"],
  },
})
```

**After** (recommended for new and refactored code):

```ts
import { definePolicy, self } from "csp-toolkit"
import csp from "vite-plugin-csp-guard"

csp({
  policy: definePolicy({
    scriptSrc: [self, "https://www.google-analytics.com"],
    fontSrc: [self, "https://fonts.gstatic.com"],
  }),
})
```

`csp-toolkit` is a **dependency** of `vite-plugin-csp-guard` (it is not inlined into the plugin’s published build; the plugin loads it from `node_modules` at runtime). You usually get a copy automatically when you install the plugin, so a direct `import` of `definePolicy` from `csp-toolkit` in your Vite config can work without adding it to your app’s `package.json`. If you want to pin or document the version you rely on, add `csp-toolkit` as a direct dependency and align the range with the plugin.

See [Creating your policy](/guides/policy) and the [API examples](/api-docs#example) for more patterns.
